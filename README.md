# Tenant Matching Prototype (Co-living) — Explainable Matching Engine + Streamlit Demo

## 0) What this project is (in simple terms)
This repository contains a working prototype that:
1) creates tenant profiles (synthetic / fake data for testing),
2) computes a compatibility score between every pair of tenants,
3) ranks the best matches for each tenant,
4) explains *why* a match is high (top drivers),
5) shows everything in a professional Streamlit web interface with live computation.

**Important scope note (thesis-relevant):**
- The current prototype is a **rule-based, deterministic matching framework** (not machine learning).
- It is built to be **transparent and explainable**, and to be a strong base for later validation (interviews/surveys and, potentially, ML in future iterations).

## 2) How the system works (step-by-step)

### Step 1 — Define the tenant profile model (variables)
Each tenant is represented by **10 variables**.
Each variable is scored on a **1–5 scale**, where:
- 1 = very low / minimal preference
- 3 = medium / neutral
- 5 = very high / strong preference

### Step 2 — Generate synthetic data (for prototyping)
Because real co-living tenant data may not be available during early research phases, the engine creates synthetic profiles.

In `matching_engine.py`, synthetic profiles are generated with:
- `N_PROFILES = 300`
- `SEED = 1234` (fixed seed → reproducible results)

Output: `synthetic_profiles_v2.csv`

### Step 3 — Compute compatibility live (in app.py)
For each selected tenant A (or custom profile in Tab 2), `app.py` compares A with every other tenant B (A ≠ B) and computes a compatibility score in percentage (0–100%).

This replaces the old CSV-based lookup. The top N matches are computed dynamically on every user interaction.

### Step 4 — Rank top matches
The app selects the **top N** best matches (configurable via sidebar slider, 1–15). Default is 5.

### Step 5 — Generate explainability
For each match, the app:
- computes per-variable contributions,
- sorts them by impact (weight × score),
- displays the top 3 "driver" variables as tags,
- generates a verbal explanation (intro + strong alignments + areas to watch).

### Step 6 — Streamlit app displays results
`app.py` displays:
- Tenant profile with segmented bars (2-column grid)
- Radar chart comparing selected tenant vs best match
- Top N match cards with donut chart, rank badge, driver tags, verbal explanation
- Expandable variable breakdown per match card (segmented bars with you/them comparison)

## 3) Tenant variables (what they mean)
The model uses the following variables (1–5 scale).

1. `cleanliness_level` — How high the person's cleanliness standard is.
2. `noise_tolerance` — How tolerant the person is to noise at home.
3. `sleep_schedule` — Sleep pattern timing (early bird ↔ night owl).
4. `routine_structure` — How structured and regular the person's daily routine is.
5. `WFH_frequency` — How often the person works from home.
6. `sociability_level` — Preference for social interaction at home.
7. `guest_tolerance` — Comfort level with guests visiting the house.
8. `privacy_need` — How much privacy and personal space the person needs.
9. `conflict_style` — How the person handles conflict (avoidant ↔ direct).
10. `shared_spaces_usage` — Intensity of using shared spaces vs staying in private space.

---

## 4) How we designed the variables (research rationale)
The variable set is designed as a **prototype-stage decision framework** for co-living.
The rationale is:

- Co-living friction typically comes from repeated daily interactions and norms:
  cleanliness, noise, sleep rhythm, privacy, guests, shared space behavior, etc.
- These 10 variables are a **structured way to represent** those friction drivers.
- The framework is intentionally simple (1–5 scale) so it is:
  - easy to understand (for non-technical stakeholders),
  - easy to validate with interviews/surveys,
  - easy to extend later.

In the thesis, these variables should be justified through:
- literature (tenant satisfaction, conflict drivers, co-living dynamics),
- qualitative validation (operator interviews),
- quantitative validation (surveys / future data).

## 5) Similarity vs Complementarity (key modeling idea)

### 5.1 Similarity variables (8)
For these, **alignment reduces friction**.
The match score increases when two tenants have similar values.

Used for: `cleanliness_level`, `noise_tolerance`, `sleep_schedule`, `routine_structure`, `WFH_frequency`, `guest_tolerance`, `privacy_need`, `conflict_style`

**Intuition:** If two people have very different cleanliness standards or sleep schedules, co-living friction increases. Similar preferences tend to work better.

### 5.2 Complementarity variables (2)
For these, a "balanced pairing" can be beneficial.
We model complementarity as a "mirror balance" logic on a 1–5 scale.

Used for: `sociability_level`, `shared_spaces_usage`

**Intuition:** Some household dynamics work well when people balance each other (e.g., not everyone is extremely social, not everyone heavily occupies shared spaces). This is a modeling assumption to be validated later.

## 6) The math: how compatibility is computed

### 6.1 Notation
- Let tenant A have value `A_f` for feature `f`
- Let tenant B have value `B_f` for feature `f`
- Values are in `{1,2,3,4,5}`
- Each feature has a weight `w_f`
- Each feature produces a normalized score `s_f(A,B)` in `[0,1]`

### 6.2 Similarity score
`s_f(A,B) = 1 - (|A_f - B_f| / 4)`

Examples:
- A=1, B=1 → score=1.00 (perfect similarity)
- A=1, B=2 → score=0.75
- A=1, B=5 → score=0.00 (max difference)

### 6.3 Complementarity score
Target sum = 6 (best complementarity when A_f + B_f ≈ 6)

`s_f(A,B) = 1 - (|A_f + B_f - 6| / 4)`

Examples:
- (1,5): sum=6 → score=1.00
- (3,3): sum=6 → score=1.00
- (1,1): sum=2 → score=0.00
- (5,5): sum=10 → score=0.00

### 6.4 Weighted aggregation
`compat(A,B) = (Σ_f w_f × s_f(A,B)) / (Σ_f w_f) × 100`

Produces a score in [0, 100].

## 7) Weights (importance assumptions, prototype stage)

| Variable | Weight | Rationale |
|---|---|---|
| `cleanliness_level` | 1.3 | Major daily friction driver |
| `noise_tolerance` | 1.3 | Major daily friction driver |
| `sleep_schedule` | 1.2 | Strong daily impact |
| `routine_structure` | 1.0 | Baseline |
| `WFH_frequency` | 0.8 | Context-dependent |
| `sociability_level` | 1.0 | Baseline |
| `guest_tolerance` | 1.0 | Baseline |
| `privacy_need` | 1.1 | Emotional comfort |
| `conflict_style` | 0.9 | Conflict probability |
| `shared_spaces_usage` | 0.9 | Partly handled by complementarity |

In the thesis, these weights should be validated with operator interviews, stress-tested via sensitivity analysis, and refined via survey evidence.

## 8) Explainability: how the "top drivers" are computed
For each match (A,B), features are ranked by contribution:

`c_f = w_f × s_f(A,B)`

The top 3 features by contribution become the "driver tags" shown on each match card.

The verbal explanation is generated in three parts:
1. **Intro paragraph** — calibrated on score (≥85% = excellent, ≥70% = good, <70% = moderate)
2. **Strong alignments** — features with score ≥ 0.75, showing exact values
3. **Areas to watch** — features with score < 0.50, with contextual note

## 9) UI components (v4)

### Radar chart
- SVG spider/star chart, 10 axes (one per variable)
- Selected tenant = teal filled polygon
- Best match = navy dashed polygon
- Vertex dots with numeric values (1–5) for selected tenant
- Canvas: 640×500px with padding to avoid label clipping
- Rendered via `streamlit.components.v1.html()`

### Segmented bars
- 5 horizontal segments per variable (one per score point)
- Teal = selected tenant / "you", grey = match / "them"
- Shown in 2-column grid (5 variables left, 5 right)
- Built as HTML string, passed to `st.markdown(unsafe_allow_html=True)` from outside helper functions

### Match cards
- Rank badge (navy pill), tenant name, compatibility score
- Top 3 driver tags
- Donut SVG (color-coded: teal ≥80%, dark teal ≥60%, amber ≥40%, red <40%)
- Expandable variable breakdown

### Critical Streamlit HTML constraint
HTML strings must use simple tags only. Complex nested divs with many inline styles often render as raw code. Helper functions must **return** HTML strings — never call `st.markdown()` internally.

## 10) How to run locally

### Generate (or regenerate) the CSV files
```bash
python3 matching_engine.py
```
Creates: `synthetic_profiles_v2.csv`, `top_matches_explained_v2.csv`

### Launch the Streamlit demo
```bash
source .venv/bin/activate
streamlit run app.py
```
Then open: `http://localhost:8501`
